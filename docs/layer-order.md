# 元素顺序与图层功能设计规则

> 状态：当前设计参考。
> 当前用途：保留元素顺序、锁定、隐藏、打印重叠提醒和数据规范化规则。

## 结论

当前“层级”功能应重新收敛为画图软件语义下的图层顺序：当多个元素在画布上重叠时，用户可以明确控制哪个元素在上方、哪个元素在下方。

不再单独建立新的图层面板。左边栏现有元素列表直接承担“元素管理 + 图层顺序”的职责：

- 元素列表越上方，画布显示越上层
- 元素列表越下方，画布显示越底层
- 新增元素默认插入到最上方
- 用户可以通过拖拽元素列表调整顺序
- 置顶、上移、下移、置底作为快捷操作保留
- 隐藏元素不显示、不打印，但仍保留在元素列表中
- 锁定元素仍显示、仍打印、仍可选中，但不可移动、缩放或删除
- 打印前检测重叠元素，只提醒，不阻止正常打印

“层级”这个词在界面中尽量移除，统一使用“元素”“元素顺序”或“图层顺序”。避免用户误解为分类层级、树状层级或权限等级。

## 当前问题

### 名称容易误导

程序内部已经使用 `zIndex` 表示画布元素叠放顺序，但界面里同时出现“层级”“图层”“元素列表”等概念。用户很难判断这些是否是同一个东西。

后续应明确：

- 对用户说“元素顺序”或“图层顺序”
- 对开发实现说 `zIndex`
- 不再把“层级”作为主要功能名称

### 列表顺序和层级含义需要绑定

用户对画图软件的直觉是：列表越靠上，元素越靠上。

如果元素列表只是对象清单，而图层顺序藏在按钮或数字中，用户会觉得调序没有反馈。元素列表应该直接成为顺序本身。

### 第几层文案不直观

“第 1 层 / 共 N 层”容易产生歧义：

- 有人会理解第 1 层是最上层
- 有人会理解第 1 层是最底层
- 列表倒序显示时，数字和视觉位置还会进一步打架

第一版建议取消第几层文案，让列表位置表达上下关系。必要时只显示“最上方”“最下方”等状态。

### 打印不是屏幕合成

画布可以通过 CSS `z-index` 做出遮挡效果，但热敏标签打印机不一定支持真正的图层合成。

如果多个元素重叠后仍按 TSPL 原生命令分别输出，可能出现：

- 重叠区域重复打印
- 黑色区域更粗、更糊或更深
- 文本、条码、二维码互相干扰
- 屏幕看似上层遮住下层，但打印机实际两个都打印

因此第一版不承诺重叠遮挡的打印效果，只做风险提示，实际效果通过测试决定后续策略。

## 新模型

### 元素顺序

文档内部仍可以用 `zIndex` 表示顺序：

- `zIndex = 0` 表示最底层
- `zIndex = N - 1` 表示最上层
- 保存、导入、删除、复制、拖拽调序后都应重新规范化为连续整数

但用户不需要看到 `zIndex` 或“第几层”。

### 元素列表

左边栏元素列表是唯一的图层管理入口。

列表显示规则：

- 列表顶部 = 最上层
- 列表底部 = 最底层
- 画布上方元素在列表中也更靠上
- 选中的画布元素在列表中高亮
- 点击列表行选中对应元素
- Ctrl 点击列表行进行多选

每个元素行建议包含：

- 元素类型图标
- 元素名称
- 简短内容预览
- 显示 / 隐藏按钮
- 锁定 / 解锁按钮
- 可拖拽手柄

不单独新增“图层面板”，避免左边栏界面被拆碎。

## 交互规则

### 新增元素

新增元素默认放到最上层，也就是元素列表顶部。

原因：

- 新对象通常是用户当前关注对象
- 符合多数编辑器直觉
- 可以立即覆盖已有对象，便于观察和调整

### 拖拽调序

元素列表支持拖拽排序。

拖拽后的结果必须立即反映到画布：

- 拖到更上方，画布上也更靠上
- 拖到更下方，画布上也更靠下
- 多选拖拽可以后置，第一版可只支持单行拖拽

### 快捷按钮

保留四个快捷操作：

- 置顶
- 上移一层
- 下移一层
- 置底

这些按钮作用于当前选中元素。

多选时，选中元素作为一个整体移动，保持它们内部相对顺序不变。

### 画布点选

点击重叠区域时：

- 默认选中最上方的可见元素
- Alt 点击时穿透选择下一层元素
- 隐藏元素不参与点选
- 锁定元素可被选中，但不可拖动、缩放或删除

### 隐藏元素

隐藏元素：

- 仍保留在元素列表中
- 不显示在画布
- 不参与画布点选
- 不参与打印
- 可以通过元素列表重新显示

### 锁定元素

锁定元素：

- 仍显示在画布
- 仍参与打印
- 仍保留在元素列表中
- 可以被选中，方便查看属性或解锁
- 不允许移动、缩放、旋转、删除

## 打印策略

### 第一版策略

第一版保持当前原生命令打印方式，不因为重叠而阻止打印。

打印前执行重叠检测：

- 检测所有可见元素的外接矩形是否相交
- 如果发现重叠，在打印确认区域或弹窗中提醒
- 提醒后仍允许用户继续正常打印

提示文案建议：

> 发现 N 组元素重叠。打印机可能会把重叠区域重复打印，导致变粗、变黑或影响扫码。请测试确认效果。

如果条码或二维码参与重叠，追加提示：

> 条码或二维码参与重叠，可能影响扫码识别。

### 不在第一版做的事

第一版暂不做：

- 自动阻止重叠打印
- 自动裁剪下层元素
- 自动合成整张位图
- 自动判断条码、二维码可扫性
- 对不同打印机做特化覆盖规则

### 后续候选方案

如果测试发现原生命令打印无法满足重叠场景，再评估“位图一致性打印模式”：

- 先把整个标签按画布效果合成为黑白位图
- 再用一次 `BITMAP` 打印整张标签

优点：

- 最接近屏幕预览
- 避免多个重叠元素分别打印造成重复输出

缺点：

- 条码、二维码变成位图后需要验证清晰度和扫码率
- 文本和线条锐利度可能受位图分辨率影响
- 失去部分打印机原生命令优势

## 数据与实现边界

### 统一顺序工具

后续实现时，应把图层顺序相关逻辑集中到一组明确函数中，避免各处手写排序。

建议命名方向：

- `sortElementsByLayer`
- `normalizeLayerOrder`
- `moveElementsToFront`
- `moveElementsForward`
- `moveElementsBackward`
- `moveElementsToBack`
- `moveElementBefore`
- `moveElementAfter`
- `getTopmostElementAtPoint`

### 数据规范化时机

以下场景都应规范化图层顺序：

- 打开模板
- 导入文件
- 新增元素
- 删除元素
- 复制元素
- 拖拽调序
- 快捷按钮调序
- 保存模板
- 打印前构建请求

规范化目标：

- `zIndex` 连续
- `zIndex` 唯一
- 顺序稳定
- 没有 `zIndex` 的旧数据可按原数组顺序补齐

## 验收标准

### 编辑体验

- 用户只看元素列表，就能判断谁在上、谁在下
- 拖动元素列表后，画布叠放立即变化
- 置顶、上移、下移、置底符合列表方向
- 新增元素出现在列表顶部并显示在最上方
- 隐藏和锁定状态清晰可见

### 数据稳定性

- 保存再打开后，元素顺序不变
- 删除元素后，剩余元素顺序不乱
- 复制元素后，新元素位置符合规则
- 导入旧数据后，顺序能被补齐并稳定保存

### 打印提醒

- 无重叠时不打扰用户
- 有重叠时能明确提醒
- 条码、二维码参与重叠时有额外风险提示
- 提醒后用户仍可继续打印
