# Initial Notes - 2026-07-04

## Source Context

These notes summarize the initial discussion that led to the YiboLabel project.

The starting point was dissatisfaction with the vendor software installed at:

`C:\Program Files (x86)\Dlabel\DLabel.exe`

## Pain Points

- the software includes too many unnecessary features
- template functionality is poor
- available templates are limited by membership tier
- the app requires online login
- the desired workflow is much simpler than the vendor product

## Desired Replacement

A standalone local printing tool with only the useful basics:

- connect to the printer
- edit labels
- save templates locally
- print labels

## Early Technical Observation

The installed software appears to include both:

- local native printing capabilities
- cloud-linked product features

Observed hints included:

- Qt-related runtime files
- local USB and print modules
- URLs for login, member pages, sync, and online template flows
- logs indicating direct USB printer interaction

This implies the project may not need to reproduce the vendor cloud stack in order to achieve practical local printing.

## Product Philosophy

YiboLabel should be:

- offline-first
- local-first
- personal-use oriented
- intentionally small in scope

## Naming Decision

The chosen project name is:

`YiboLabel`

Reasoning:

- consistent with the user's existing `Yibo` software naming pattern
- clear enough to describe the project purpose
- suitable for product name, repository name, and executable naming

## Suggested Naming Conventions

- product name: `YiboLabel`
- repository name: `yibo-label`
- executable name: `YiboLabel.exe`

## Early Constraint Preference

When the architecture allows choice, default to:

- `http` over `websocket`

Only use `websocket` if a later requirement clearly makes it necessary.

## First Strategic Question

The most important technical problem to solve is the print path:

- how the target printer is detected
- how print data is sent
- whether Windows system printing is enough
- whether direct USB communication is required

Everything else should be kept secondary until that path is understood.
