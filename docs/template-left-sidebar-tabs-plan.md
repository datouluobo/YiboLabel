# 模板左侧栏三标签方案与实施计划

## 目标

当前模板编辑页左侧区域同时承载：

- 添加元素
- 元素列表
- 绑定
- 候选

现状的问题是：

- 信息层级分散，用户需要在顶部和左栏之间来回切换
- “绑定”和“候选”与“元素”同属模板内容工作区，却分散在不同位置
- 左栏职责不够稳定，难以形成清晰心智

这份文档用于固定新的左侧栏方案，作为后续 UI 调整与实现依据。

## 方案摘要

模板编辑页左侧栏统一重构为三标签工作区：

- 元素
- 分组
- 条目

页面结构上，左栏不再把“绑定”“候选”作为顶部独立入口，而是作为左栏内部工作模式统一承载。

最终目标是让用户围绕一条稳定主链路完成配置：

1. 在“元素”中新增或选中模板元素
2. 在“分组”中为当前元素建立词库分组关系
3. 在“条目”中查看和选择具体条目内容

这份文档中的内容以定稿方案为准，后续实现、评审和交互细节补充均应优先遵循这里的边界。

## 已确认结论

左侧栏改为三标签结构：

- 元素
- 分组
- 条目

命名原则：

- “元素”表示模板里的对象本体
- “分组”对应词库中的分组层
- “条目”对应词库中的条目层

词库页术语同步使用：

- 分组
- 条目

不再使用：

- 绑定
- 候选
- 选库
- 选词

原因是这些名称更偏动作或阶段，容易让用户误解为“先选词库，再选内容”，不如直接对齐词库结构清楚。

## 适用范围

本方案覆盖：

- 模板编辑页左侧栏
- 与左栏联动的画布绑定关系显示
- 词库相关术语在模板编辑页中的呈现

本方案不覆盖：

- 词库页整体信息架构重做
- 分组和条目的后台数据模型
- 多选元素的批量高级操作
- 复杂动画和视觉皮肤

## 设计原则

### 1. 左栏只承载一类事情

左栏统一定义为“模板内容工作区”。

三标签分别回答三个问题：

- 元素：模板里有哪些对象，如何新增和整理
- 分组：当前对象对应词库中的哪类内容
- 条目：当前对象可使用哪些具体内容

### 2. 固定区只放上下文与关键操作

左栏宽度有限，固定区只保留：

- 当前上下文
- 搜索
- 关键开关
- 关键操作

长列表全部放入滚动区。

### 3. 统一结构，降低学习成本

三个标签都采用同一骨架：

- 顶部固定区
- 下方滚动区

用户理解其中一个标签后，应能快速理解另外两个标签。

### 4. 词库结构优先于动作命名

“分组”和“条目”直接对齐词库结构，避免用户额外理解“绑定”“候选”等中间概念。

## 三标签内容方案

## 页面级统一规则

三个标签统一遵循以下规则：

- 标签栏固定在左栏顶部
- 每个标签采用“固定顶部区 + 滚动内容区”
- 固定顶部区只放上下文、搜索和关键操作
- 列表、长内容、折叠块全部进入滚动区
- 不允许整块左栏跟随内容一起滚动

这样可以确保：

- 标签切换后位置稳定
- 用户不会丢失“当前元素 / 当前分组”上下文
- 左栏在窄宽度和长列表下仍可控

## 元素

### 职责

元素标签用于处理模板对象本体。

它解决的问题是：

- 如何新增元素
- 当前模板有哪些元素
- 元素如何排序和整理

### 固定顶部

- 6 个添加元素按钮
- `元素` 标题
- 元素总数
- 排序操作
  - 置顶
  - 上移
  - 下移
  - 置底

可选保留：

- 元素搜索
- 仅看可见元素
- 仅看未绑定元素

### 滚动区域

- 元素列表

元素项建议显示：

- 元素类型图标
- 元素名称或内容摘要
- 显示/隐藏状态
- 锁定状态
- 是否已绑定分组

### 不放入此标签的内容

- 分组列表
- 条目列表
- 词库搜索
- 绑定关系显示控制

这样可以保持“元素”标签的职责纯净。

## 分组

### 职责

分组标签用于为当前元素建立词库分组关系。

它解决的问题是：

- 当前元素绑定了哪些分组
- 可以绑定哪些分组
- 如何把分组拖放到元素上建立关系
- 如何在画布上查看绑定关系

### 固定顶部

- 当前元素信息
  - 例如：`当前元素：物料编码`
- 绑定关系显示开关
- 显示范围切换
  - 选中元素
  - 所有元素
- 简短拖放提示
  - 例如：`拖动分组到画布元素可建立绑定`
- 分组搜索框

推荐把“绑定关系显示”设计为一套紧凑控件：

- `显示绑定关系`
- 范围：`选中元素 / 所有元素`

不建议做成两个独立的大按钮，以免占用横向空间。

### 滚动区域

- 分组列表

分组项建议显示：

- 分组名
- 条目数
- 是否已绑定到当前元素

若当前元素已绑定该分组，列表项需要有明确选中态。

### 额外功能

#### 1. 在画布上显示绑定关系

开启显示后，在画布元素框体旁显示其绑定分组。

显示策略建议：

- 默认关闭，避免常态干扰画布
- 支持只显示当前选中元素的绑定关系
- 支持显示所有元素的绑定关系

用途：

- 方便用户快速检查当前模板配置是否完整
- 方便拖放绑定前确认目标对象

#### 2. 分组拖放到画布元素

支持从分组列表直接拖放到画布元素上建立绑定关系。

推荐规则：

- 拖到某个元素即为该元素添加该分组
- 若分组已绑定，则给出已绑定反馈，不重复添加
- 拖放是快捷方式，仍保留点击绑定的常规方式

这样兼顾效率与可发现性。

### 不放入此标签的内容

- 完整条目列表
- 添加元素按钮
- 元素排序按钮

分组标签只负责“选类别”和“建关系”，不负责处理具体条目。

## 条目

### 职责

条目标签用于查看和选择词库中的具体条目。

它解决的问题是：

- 当前分组下有哪些条目
- 当前元素可以使用哪些具体内容
- 是否需要跨分组查看条目

### 固定顶部

- 当前元素信息
- 当前分组信息
- 条目搜索框
- 视图切换
  - 当前分组
  - 全部分组

如果当前元素尚未绑定分组，顶部应明确显示空状态提示。

### 滚动区域

- 条目列表

### 默认模式

默认使用“当前分组”模式。

原因：

- 条目操作可能高频，但通常是集中处理
- 直接显示当前分组条目最利于快速选择
- 左栏宽度有限，不适合默认铺开所有分组内容

### 全部分组模式

支持切换到“全部分组”模式。

此模式下采用分组折叠结构：

- 默认全折叠
- 当前分组自动展开
- 搜索命中的分组自动展开
- 其它分组保持折叠

这样既保留跨分组浏览能力，又不会让左栏在窄宽度下失控。

### 条目项建议显示

- 条目值
- 来源分组

当处于“当前分组”模式时，来源分组可弱化显示；当处于“全部分组”模式时，来源分组应更明确。

### 不放入此标签的内容

- 分组主操作
- 元素新增和排序
- 大段说明文字

条目标签应强调高效浏览与快速选择。

## 固定区与滚动区总表

### 元素

固定顶部：

- 添加元素按钮
- 标题与数量
- 排序操作

滚动区域：

- 元素列表

### 分组

固定顶部：

- 当前元素信息
- 绑定关系显示开关
- 显示范围切换
- 拖放提示
- 分组搜索

滚动区域：

- 分组列表

### 条目

固定顶部：

- 当前元素信息
- 当前分组信息
- 条目搜索
- 视图切换

滚动区域：

- 条目列表
- 在“全部分组”模式下使用自动折叠分组

## 宽度与信息密度策略

分组和条目迁移到左栏后，必须优先控制窄栏可读性。

原则如下：

- 标签名保持短字数：元素 / 分组 / 条目
- 列表项以单行主信息为主
- 顶部操作尽量使用紧凑控件
- 搜索框优先保留全宽
- 提示文案只保留一句

具体建议：

- 分组项显示为“名称 + 条数”
- 条目项显示为“条目值”，必要时辅以轻量来源信息
- 不在左栏顶部堆叠多行说明文字
- 不把多个长按钮横向并排

## 低保真布局建议

以下布局建议只用于明确结构，不限定最终视觉样式。

### 左栏整体骨架

左栏建议固定为四层：

1. 标签栏
2. 标签内固定顶部区
3. 标签内滚动内容区
4. 与画布之间的自然分隔边界

建议规则：

- 标签栏始终固定在最上
- 切换标签时，顶部区结构尽量稳定
- 滚动只发生在内容区，不让整个左栏一起滚

这样可以避免用户在不同标签里频繁失去位置感。

建议从上到下的视觉节奏为：

- 标签切换
- 当前位置上下文
- 当前标签操作
- 当前标签主列表

不要把说明性信息放到比操作更靠前的位置。

### 标签栏

建议使用三等分标签：

- 元素
- 分组
- 条目

布局建议：

- 三个标签等宽
- 文案单行居中
- 当前标签使用明确高亮
- 不在标签上叠加复杂状态徽标

如需提示数量，优先放进标签内容页标题，而不是直接塞进标签页签。

标签栏的定稿要求：

- 不显示副标题
- 不显示多余图标
- 不显示角标红点

保持最小复杂度，避免再次变成新的信息堆叠区。

### 元素标签布局

建议布局顺序为：

1. 添加区
2. 元素标题行
3. 排序操作行
4. 元素列表

#### 1. 添加区

建议保持 2 列 3 行宫格：

- 文本
- 条码
- 二维码
- 线条
- 矩形
- 图片

建议：

- 每个按钮保持图标 + 名称
- 不建议压成 1 行横排
- 不建议在这里放说明文字

原因：

- 这里是低频理解、高频点击的区域
- 2 列 3 行更稳，也更适合中英文和未来扩展

定稿要求：

- 这 6 个按钮只出现在“元素”标签中
- 不复制到顶部全局工具条
- 不与“分组”“条目”联动显示

#### 2. 元素标题行

建议包含：

- 左侧：`元素`
- 右侧：元素数量

可选附加：

- 一个轻量帮助图标

不建议在这一行放太多开关，否则会挤压标题辨识度。

#### 3. 排序操作行

建议保持单行横排：

- 置顶
- 上移
- 下移
- 置底

如果宽度不足，优先方案是：

- 保留图标
- 缩短按钮内边距

不建议把这四个按钮拆成两行，否则会把元素列表整体下推过多。

#### 4. 元素列表

建议单列纵向列表。

每个元素项建议分成三段：

- 左：拖拽柄 + 类型图标
- 中：名称或内容摘要
- 右：状态操作

右侧状态操作建议保留：

- 显示/隐藏
- 锁定/解锁

若需要显示“已绑定分组”，建议使用轻量状态点或小徽标，不要展开成长文案。

元素列表的定稿要求：

- 列表优先服务于定位和整理
- 不在元素列表中直接展开分组详情
- 不在元素列表中直接展开条目内容

### 分组标签布局

建议布局顺序为：

1. 当前元素上下文
2. 绑定关系显示控制
3. 拖放提示
4. 搜索框
5. 分组列表

#### 1. 当前元素上下文

建议固定成一行或两行紧凑信息：

- 标题：`当前元素`
- 值：元素名称

如果元素名称较长：

- 单行截断
- hover 或详情态再显示全名

这里的作用是让用户始终知道“现在在给谁配分组”。

#### 2. 绑定关系显示控制

建议做成一行紧凑组合控件：

- 左：`显示绑定`
- 右：范围切换 `选中 / 全部`

显示逻辑建议：

- 默认关闭
- 开启后默认显示“选中”
- 切到“全部”时，需有更明显的激活反馈

不建议使用两颗独立按钮：

- 显示选中元素绑定
- 显示所有元素绑定

因为在窄栏中会显得冗长且语义冲突。

定稿要求：

- 开关入口固定留在“分组”标签
- 不复制到“元素”或“条目”标签
- 当前会话中的显示状态可以临时记忆

#### 3. 拖放提示

建议是一条短说明，弱化显示：

- `拖动分组到画布元素可建立关联`

建议放在显示控制和搜索框之间。

作用：

- 用户第一次进入此标签时能发现拖放能力
- 但不抢分组列表的注意力

#### 4. 搜索框

建议全宽。

占位文案建议类似：

- `搜索分组`

不建议加多余前缀，如“按名称搜索分组”，会显得拥挤。

#### 5. 分组列表

建议单列纵向列表。

每个分组项建议分成两层信息：

- 主行：分组名
- 次行：条目数或其它轻量辅助信息

如果需要压缩得更紧，也可以合并成单行：

- `发货编码  3条`

分组项状态建议：

- 已绑定到当前元素：明显选中态
- 可拖放：鼠标悬停有拖动态提示
- 搜索命中：按常规高亮

分组列表的定稿要求：

- 点击分组项可建立或切换当前元素与分组的关系
- 拖放是增强能力，不是唯一入口
- 分组列表不直接展开内部全部条目

### 条目标签布局

建议布局顺序为：

1. 当前元素上下文
2. 当前分组上下文
3. 搜索框
4. 视图切换
5. 条目列表

#### 1. 当前元素上下文

与“分组”标签保持相同样式。

目的是保持标签切换时的稳定感，不让用户重新学习。

#### 2. 当前分组上下文

建议放在当前元素下方，用于明确条目来源。

建议样式：

- 标题：`当前分组`
- 值：分组名

如果当前元素未绑定分组：

- 直接在此区域显示空状态提示
- 并弱化下面条目列表区域

#### 3. 搜索框

建议全宽。

占位文案建议：

- `搜索条目`

如支持按内容筛选，可写：

- `按内容筛选`

#### 4. 视图切换

建议使用两段式切换：

- 当前分组
- 全部分组

布局建议：

- 放在搜索框下方
- 单独占一行

原因：

- 搜索和视图切换是两类动作
- 如果强行挤在一行，窄栏里可读性会明显变差

#### 5. 条目列表

##### 当前分组模式

建议直接平铺条目。

每项只显示：

- 条目值

如条目值可能很长：

- 单行显示
- 超出截断
- 选中或悬停时显示完整内容

##### 全部分组模式

建议使用折叠分组列表。

每组结构：

- 分组标题
- 条目数
- 展开/折叠箭头

展开规则建议：

- 默认全部折叠
- 当前分组默认展开
- 搜索命中的分组自动展开
- 用户手动展开的组，在本次停留期间保持状态

这样可以兼顾总览能力与滚动效率。

条目标签的定稿要求：

- 默认进入“当前分组”模式
- “全部分组”模式必须为折叠式
- 不允许默认全展开所有分组条目

## 关键交互细化建议

### 元素到分组的切换

当用户在“元素”里选中一个元素后：

- 切到“分组”时应自动带入该元素上下文
- 不要求再次选择目标元素

若没有选中元素：

- “分组”和“条目”都应优先显示空状态引导

### 分组绑定建立方式

建议至少支持两种方式：

- 点击分组项建立或切换绑定
- 拖放分组到画布元素建立绑定

推荐优先级：

- 点击方式作为基础能力
- 拖放方式作为高效增强能力

定稿规则：

- 点击与拖放应产出同等绑定结果
- 不允许两种方式产生不同的数据语义

### 绑定关系画布显示

建议显示位置：

- 元素框体外侧
- 靠近元素标题或边缘

建议显示内容：

- 分组名称

如果一个元素绑定多个分组：

- 优先单行紧凑展示
- 超出时使用 `+N` 汇总

避免在画布中展开成长列表，影响编辑视野。

定稿规则：

- 绑定关系显示属于辅助检查层，不改变元素本体布局
- 关闭显示后，画布应回到纯编辑视图

### 条目视图切换

建议记住用户本次会话最后一次选择：

- 若用户停留在“全部分组”
- 再次切回“条目”时继续保持

但若当前元素没有绑定分组：

- 仍然优先落回更易理解的空状态

### 搜索行为

建议三个原则：

- 搜索输入即时生效
- 搜索仅作用于当前标签
- 切换标签时不共享搜索词

原因：

- 三个标签的数据结构不同
- 共享搜索会增加理解成本

定稿规则：

- 搜索词不跨标签继承
- 搜索状态在切换标签后可清空，也可按标签分别记忆
- 但不能出现一个标签的搜索条件影响另一个标签结果

## 交互链路

推荐用户主流程为：

1. 在“元素”里新增或选中一个元素
2. 切到“分组”为该元素建立词库分组关系
3. 切到“条目”查看和选择具体内容

这条链路对应：

- 对象本体
- 结构映射
- 具体内容

后两个标签都应依赖“当前选中元素”这一上下文。

## 空状态建议

为了避免切换标签后用户发懵，需要给出明确空状态。

### 分组

当未选中元素时：

- 提示先在“元素”中选择一个元素

当词库中暂无分组时：

- 提示前往词库页创建分组

### 条目

当未选中元素时：

- 提示先选择元素

当当前元素未绑定分组时：

- 提示先在“分组”标签建立关联

当分组存在但无条目时：

- 提示前往词库页补充条目

## 视觉与反馈建议

这部分不限定最终视觉风格，只固定反馈方向。

### 选中态

- 当前标签要有稳定高亮
- 当前元素要有稳定高亮
- 当前分组要有稳定高亮
- 当前条目要有稳定高亮

### 悬停态

- 列表项悬停应明确可点击
- 可拖放分组项悬停时可增加拖拽提示

### 拖放反馈

- 拖动分组时，目标元素应出现接收高亮
- 放下成功后应立即反馈绑定完成
- 已绑定时应反馈“已存在”而不是静默失败

### 空状态

- 空状态文案尽量短
- 空状态优先告诉用户下一步做什么
- 不使用大段解释文字

## 研发实施输出物建议

为了减少返工，建议每个阶段都产出可复用的中间结果。

### 第一阶段输出物

- 左栏三标签命名替换清单
- 词库相关术语替换清单
- 页面信息架构对照表

### 第二阶段输出物

- 左栏结构重排后的可运行页面
- 三标签固定区与滚动区的基础样式
- 左栏宽度下的基本溢出处理

### 第三阶段输出物

- 分组绑定关系显示
- 分组拖放建立绑定
- 点击建立绑定的统一行为

### 第四阶段输出物

- 条目标签“当前分组”模式
- 条目标签“全部分组”模式
- 自动折叠和搜索展开规则

### 第五阶段输出物

- 空状态文案
- 提示反馈
- 状态样式补齐
- 交互验收修正项

## 评审重点

在进入编码前或阶段交付后，建议围绕以下问题评审：

1. 用户是否能在 3 秒内看懂三个标签分别做什么
2. “元素 -> 分组 -> 条目”的链路是否比现状更顺
3. 左栏顶部固定区是否过高，是否挤压了主列表
4. 分组拖放是否真的提升效率，而不是只是增加花样
5. 条目“全部分组”模式是否保持了清晰而非信息爆炸
6. 画布显示绑定关系时是否仍保留足够编辑视野

## 验收建议

方案实施后，建议至少检查以下结果：

1. 用户能否一眼理解三标签的职责差异
2. 用户能否在不看说明的情况下完成“选元素 -> 绑分组 -> 找条目”
3. 左栏在较窄宽度下是否仍保持清晰
4. 分组拖放是否比原有绑定方式更高效
5. 条目“全部分组”模式是否可用而不拥挤
6. 画布上的绑定关系显示是否有帮助而不过度打扰

## 实施计划

## 第一阶段：信息架构与命名落地

目标：

- 统一左栏三标签命名
- 固定左栏职责边界
- 统一词库页相关术语

任务：

- 左栏入口从“绑定 / 候选”调整为“分组 / 条目”
- 第一标签定义为“元素”，包含添加区和元素列表
- 词库页文案统一使用“分组 / 条目”
- 对现有相关按钮、标题、提示文案做一次术语盘点

交付结果：

- 页面结构和术语统一
- 后续实现不再围绕旧名称扩展

## 第二阶段：三标签布局重构

目标：

- 把左栏改造成统一的“固定顶部 + 滚动列表”结构

任务：

- 重组“元素”标签中的添加区和列表区
- 为“分组”标签增加当前元素上下文和搜索区
- 为“条目”标签增加当前元素、当前分组、搜索和视图切换区
- 控制顶部区域高度，给列表保留最大滚动空间

交付结果：

- 左栏结构稳定
- 三标签视觉骨架统一

## 第三阶段：分组关系可视化与拖放

目标：

- 让分组关系建立和检查更直接

任务：

- 增加“显示绑定关系”开关
- 支持范围切换：选中元素 / 所有元素
- 在画布元素框体旁显示分组信息
- 支持从分组列表拖放到画布元素建立绑定关系
- 保留非拖放的常规绑定方式

交付结果：

- 用户可快速检查模板绑定情况
- 建立绑定关系的效率提高

## 第四阶段：条目浏览模式完善

目标：

- 在窄左栏里兼顾高效选取与全局浏览

任务：

- 默认实现“当前分组”条目模式
- 增加“全部分组”视图
- 在“全部分组”视图中实现自动折叠
- 定义搜索命中后的自动展开规则

交付结果：

- 条目标签既适合高频集中使用，也支持跨分组浏览

## 第五阶段：空状态与细节打磨

目标：

- 降低首次使用和切换标签时的迷惑感

任务：

- 补齐未选中元素时的提示
- 补齐未绑定分组时的提示
- 补齐无分组、无条目时的提示
- 优化列表项状态表达
- 优化拖放反馈和绑定成功反馈

交付结果：

- 整体体验更稳定
- 用户更容易理解当前所处步骤

## 风险与控制点

### 风险 1：顶部固定区过高

如果“分组”或“条目”顶部塞入过多说明、按钮和状态，滚动区会被明显压缩。

控制方式：

- 固定区只保留核心上下文与操作
- 所有解释性信息压缩为一行提示
- 不增加次要按钮排

### 风险 2：左栏职责再次混乱

如果后续把条目详情、分组管理、元素附加工具都塞回左栏，三标签边界会再次模糊。

控制方式：

- 元素只管对象本体
- 分组只管关系
- 条目只管具体内容

### 风险 3：条目“全部分组”模式过载

如果默认展开或展示过多附加信息，条目标签会迅速变乱。

控制方式：

- 默认使用“当前分组”
- “全部分组”默认折叠
- 条目项只保留主信息

### 风险 4：拖放能力可发现性不足

如果只做拖放而不提供清楚提示，很多用户不会意识到这项能力存在。

控制方式：

- 保留点击绑定作为基础路径
- 在“分组”标签固定区提供一行提示
- 对可投放目标提供清晰悬停反馈

## 实施顺序建议

建议按以下顺序推进：

1. 命名和信息架构先定稿
2. 左栏三标签骨架先落地
3. 分组标签的关系显示与拖放能力再接入
4. 条目标签的“全部分组”折叠视图后补
5. 最后统一处理空状态和细节反馈

原因：

- 先定结构，避免后续交互功能反复返工
- 先保证标签职责清晰，再叠加高级交互
- “全部分组”模式属于增强能力，可后置

## 暂不展开的内容

当前方案阶段先不展开：

- 分组和条目的批量管理
- 词库页之外的高级筛选
- 多选元素的批量绑定策略
- 跨标签的复杂联动动画

这些能力可以在三标签结构稳定后再评估是否需要。

## 最终结论

模板编辑页左侧栏后续统一采用三标签结构：

- 元素
- 分组
- 条目

三标签统一遵循：

- 顶部固定上下文和关键操作
- 下方滚动列表内容

其中：

- 元素负责对象本体
- 分组负责词库类别关系
- 条目负责具体内容浏览与选择

这是当前最清晰、最稳定、最适合后续扩展的方向。
