# 可复用视觉反馈点备忘

这份文档只沉淀目前已经验证过、以后“真有需要时可以直接复用”的 3 个点。

当前结论：

- 先记录，不立即扩散到其它流程。
- 只有当目标页面同样存在“问题多、需要定位、需要跳修复入口”的场景时，才考虑复用。

## 1. 问题优先的单开结构

适用前提：

- 一个页面下会出现多个分类。
- 用户通常只会同时处理一个分类。

可复用点：

- 默认只展开最需要处理的分类。
- 用户打开当前分类时，自动折叠其它分类。
- 通过项默认弱化，只在用户主动查看时展开。

价值：

- 明显降低信息噪音。
- 避免“所有卡片都开着”的长页滚动负担。

不适合复用的场景：

- 用户需要并排比对多个分类内容。
- 分类之间存在强依赖，需要同时展开查看。

## 2. 列表与预览的双向定位

适用前提：

- 页面同时存在“问题列表”和“空间预览/画布/示意图”。

可复用点：

- hover 列表项时，预览区临时高亮对应对象。
- click 列表项时，滚动并锁定当前问题。
- click 预览标注时，反向定位到列表项。

价值：

- 用户不需要在“看说明”和“找位置”之间来回猜。
- 特别适合布局、越界、重叠、空内容这类有空间坐标的问题。

不适合复用的场景：

- 问题本身没有空间位置，例如纯全局状态、权限、网络或保存状态。

## 3. 当前焦点的强化反馈

适用前提：

- 页面问题数量超过 3 条。
- 用户需要连续处理“上一条 / 下一条”。

可复用点：

- 当前问题自动滚动到视口内，必要时居中。
- 当前分类和当前条目都给出短时高亮反馈。
- 保留轻量导航，只在当前筛选范围内切换。

价值：

- 用户不会在长列表里丢焦点。
- 连续修问题时节奏更顺，不容易“点到了但没看见”。

不适合复用的场景：

- 页面只有 1-2 个状态项。
- 列表非常短，额外导航反而增加负担。

## 后续复用判断标准

只有同时满足下面至少 2 条，才建议真的落地复用：

- 存在 5 条以上状态项。
- 状态项能按分类组织。
- 用户需要频繁定位到具体对象或区域。
- 当前流程已经出现“找不到问题在哪”或“滚动迷路”的反馈。

如果只是“也许以后能用”，那先停在这份文档，不继续实现。
