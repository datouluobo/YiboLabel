# 模板编辑器词库动作 UX 定稿

更新日期：2026-07-20

## 文档目的

这份文档用于归档模板编辑器中“词库分组/条目”与“画布元素”之间的动作交互定稿。

覆盖范围：

- 词库页底部动作区
- 词库拖放到画布元素
- 绑定、解绑、应用、撤回的语义边界
- 拖放与按钮操作的统一动作模型
- 相关视觉反馈与通知

不覆盖：

- 词库数据模型重构
- 词库页整体信息架构重做
- 更大范围的模板编辑器布局调整
- 多元素批量词库操作

## 设计原则

### 1. 绑定与应用必须分开

两个动作的职责不同，不能混用：

- 绑定分组：建立“元素 <- 分组”的关系，不改变元素当前显示内容
- 应用条目：把条目文本写入元素内容，不改变绑定关系

### 2. 当前上下文动作与最近动作撤回必须分开

两个入口解决的是两类不同问题：

- 当前上下文动作：用户现在想对“当前选择”做什么
- 最近动作撤回：用户刚才做错了，想退回最近一次词库动作

两者不能再混成一个“撤消”概念。

### 3. 拖放与按钮共用动作实现，但不强行共用状态来源

底层动作必须统一：

- `bindGroupToElement(elementId, groupId)`
- `unbindGroupFromElement(elementId, groupId)`
- `applyEntryToElement(elementId, text, groupId)`
- `undoLastLexiconAction(actionRecord)`

但上下文来源不同：

- 拖放：以鼠标命中的目标元素为准
- 左按钮：以当前选择元素为准
- 右按钮：以最近一次词库动作记录为准

### 4. 连续拖放优先，避免视觉跳动

拖放成功后，不自动把目标元素切换成当前视觉选中元素。

原因：

- 连续大量拖放时会造成选中框频繁跳动
- 会打断用户现有选择上下文
- 会让左按钮的语义变得不稳定

## 核心状态定义

词库动作 UX 只依赖下面三类状态：

### 当前选择元素

用户当前在画布中真实选中的元素。

用途：

- 左按钮判断
- 当前关系判断

### 最近拖放目标元素

最近一次拖放命中的元素。

用途：

- 拖放过程中的视觉提示
- 拖放动作执行时的目标

限制：

- 不作为左按钮判断依据
- 不直接改变当前视觉选中元素

### 最近一次词库动作记录

最近一次成功执行的词库动作。

建议记录字段：

- `kind`: `bind-group` | `unbind-group` | `apply-entry`
- `elementId`
- `elementName`
- `groupId`
- `groupName`
- `entryText`
- `previousValue`
- `nextValue`
- `createdAt`
- `source`: `drag-drop` | `button`

用途：

- 右按钮启用与文案
- 右按钮执行撤回
- 动作完成通知

## 底部动作区定稿

底部动作区固定只占两个按钮位，不增加高度，不根据动作数量扩容。

### 左按钮

定义：

- 当前选择动作

判断范围：

- 当前选择元素
- 当前词库选择

不看：

- 最近拖放目标元素
- 最近一次词库动作记录

### 右按钮

定义：

- 撤回最近一次词库动作

判断范围：

- 最近一次词库动作记录

不看：

- 当前选择元素
- 当前词库选择
- 最近拖放目标元素

## 左按钮规则

### 词库当前选中分组，且当前选择元素存在

根据两者当前关系判断：

- 未绑定：显示 `绑定分组`
- 已绑定：显示 `取消绑定`

动作语义：

- `绑定分组`：建立当前选择元素与当前选择分组之间的绑定关系
- `取消绑定`：解除当前选择元素与当前选择分组之间的绑定关系

限制：

- 取消绑定只允许对“当前选择元素 + 当前选择分组”生效
- 不允许只选分组就做批量解绑

### 词库当前选中条目，且当前选择元素存在

显示：

- `应用条目`

动作语义：

- 把当前条目文本写入当前选择元素

### 条件不足

按钮直接禁用。

规则：

- 不显示长提示文案
- 不因为用户当前浏览行为频繁改按钮文字
- 保留上一次动作名，直到下一次进入可用状态时再随按钮一起改变

目的：

- 避免一个不可用按钮仍在后台跟着用户操作频繁变文案
- 减少视觉噪音和误导

## 右按钮规则

右按钮永远只看最近一次成功词库动作。

### 最近动作为绑定分组

显示：

- `撤消绑定`

执行结果：

- 解除刚才那次建立的绑定关系

### 最近动作为取消绑定

显示：

- `恢复绑定`

执行结果：

- 恢复刚才那次被取消的绑定关系

### 最近动作为应用条目

显示：

- `撤消应用`

执行结果：

- 把当时目标元素恢复到应用前的值

### 没有最近动作记录

按钮禁用。

## 拖放规则定稿

### 拖分组到元素

动作语义：

- 绑定分组

执行规则：

- 使用鼠标命中的目标元素
- 不要求该元素当前已经被选中
- 成功后记录最近一次词库动作
- 激活右按钮
- 不自动改变当前视觉选中元素

### 拖条目到元素

动作语义：

- 应用条目

执行规则：

- 使用鼠标命中的目标元素
- 不要求该元素当前已经被选中
- 成功后记录最近一次词库动作
- 激活右按钮
- 不自动改变当前视觉选中元素

## 拖放前视觉判断

拖放前必须先判断目标是否可接受，并直接显示将执行的动作或禁止原因。

### 分组拖放

分组拖放不检查分组内所有条目是否合法。

原因：

- 会增加复杂度
- 条目多时问题难以定位
- 会破坏“分组拖到元素 = 建立绑定关系”的动作纯度

规则：

- 拖到文本、条码、二维码元素：显示 `绑定分组`
- 拖到不支持词库的元素：显示 `不可绑定：该元素不支持词库`

### 条目拖放

#### 拖到文本元素

显示：

- `应用到文本`

#### 拖到二维码元素

显示：

- `应用到二维码`

二维码允许：

- 汉字
- 多行文本

#### 拖到条码元素

当前第一批适配按常用 Code128 规则处理。

允许：

- 单行内容
- ASCII 32-126 字符

禁止时显示：

- `不可应用：内容为空`
- `不可应用：条码不支持换行`
- `不可应用：Code128 不支持该字符`

#### 拖到非内容元素

显示：

- `不可应用：该元素不支持文本`

### 拖放高亮

建议视觉反馈：

- 可执行动作：绿色或青色高亮
- 不可执行动作：红色高亮
- 提示文案直接显示动作或禁止原因

## 按钮与拖放关系定稿

### 统一点

按钮和拖放必须调用同一套底层动作实现，并写入同一类最近动作记录。

### 区别点

按钮和拖放的目标来源不同：

- 左按钮：当前选择元素
- 拖放：鼠标命中元素

因此：

- 底层动作模型必须打通
- 上层状态来源不强行统一

这是本轮最终确认的边界。

## 动作完成通知

按钮副文案不作为本轮 UX 方案的一部分。

原因：

- 会牵连按钮高度、节奏、截断和三标签统一性
- 成本较高，不一定更好理解

本轮改为强化动作完成通知。

建议通知文案：

- `已绑定分组：发货编码`
- `已取消绑定：发货编码`
- `已恢复绑定：发货编码`
- `已应用条目：PKG-000001`
- `已撤消应用：已恢复原内容`

如果需要更明确，可在通知中附带目标元素名，例如：

- `已绑定分组：发货编码 -> 条码1`

## 本轮明确不做

- 因拖放成功自动切换当前视觉选中元素
- 通过按钮副文案显示元素与分组关系
- 只选分组时批量取消绑定
- 多元素批量取消绑定
- 复杂多层应用历史栈

## 最终结论摘要

1. 底部动作区固定两个按钮位
2. 左按钮只服务当前选择
3. 右按钮只服务最近词库动作撤回
4. 拖放成功不自动改当前视觉选中元素
5. 绑定与应用严格分离
6. 按钮与拖放共用动作实现，但不强行共用状态来源
7. 取消绑定严格限定为“当前选择元素 + 当前选择分组”
8. 右按钮在最近动作为取消绑定时，文案使用 `恢复绑定`
