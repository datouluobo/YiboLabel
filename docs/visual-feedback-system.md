# 常规视觉反馈系统

## 目标

视觉反馈系统的目标不是增加界面装饰，而是帮助用户在最短时间内理解：

- 当前状态是什么
- 是否需要处理
- 问题严重程度如何
- 下一步应该做什么

这套规则不仅适用于打印检查，后续也可复用于模板、导出、词库、设备状态等其它模块。

## 核心原则

### 1. 语义优先

同一种状态，在不同模块中应尽量保持同一语义和同一视觉方向。

例如：

- `通过` 不应在一个页面是绿色，在另一个页面变成灰色主态
- `未通过` 不应一会表示阻塞，一会只表示轻提醒

### 2. 问题优先

界面默认应该优先展示需要处理的内容，而不是把所有状态同权呈现。

意味着：

- 问题项优先展示
- 通过项默认弱化
- 大数量时先做摘要，再按需展开

### 3. 层级清晰

视觉反馈至少要区分三层：

- 总体状态
- 分类或模块状态
- 具体项目状态

用户应能先看到总判断，再逐层进入细节。

### 4. 反馈必须可行动

反馈不应停留在“告诉你有问题”，还应帮助用户理解：

- 为什么是问题
- 会造成什么影响
- 去哪里修复

### 5. 保持克制

颜色、描边、图标、动画都应服务于状态传达。

避免：

- 同屏同时出现过多高饱和警示色
- 每条提示都使用强烈动画
- 把所有项目默认展开

## 状态分级

建议统一使用三档状态：

- `通过`
- `提醒`
- `未通过`

## 三档状态定义

### 通过

含义：

- 当前条件满足
- 不阻塞流程
- 无需立即处理

视觉建议：

- 绿色系
- 低噪音背景
- 清晰但不过度强调

交互建议：

- 默认不抢主焦点
- 可在摘要层显示数量
- 明细可折叠隐藏

### 提醒

含义：

- 当前可继续操作
- 但存在风险、隐患或建议优化项

视觉建议：

- 琥珀/黄色系
- 比通过更显眼，但低于未通过

交互建议：

- 在摘要层可见
- 明细默认可折叠
- 如数量较多，应支持问题优先浏览

### 未通过

含义：

- 当前不应继续执行关键动作
- 属于阻塞项

视觉建议：

- 红色系
- 边框、图标、标题层级上给予更高强调

交互建议：

- 默认展开
- 优先排序
- 与主操作按钮状态联动

## 状态聚合规则

### 项目 -> 分类

- 只要分类内存在 `未通过`，分类状态就是 `未通过`
- 没有 `未通过` 但有 `提醒`，分类状态就是 `提醒`
- 全部通过，分类状态就是 `通过`

### 分类 -> 页面

- 只要任一分类为 `未通过`，页面总状态就是 `未通过`
- 没有 `未通过` 但存在 `提醒`，页面总状态就是 `提醒`
- 所有分类都通过，页面总状态就是 `通过`

这样可以保证全局状态与局部状态一致。

## 信息组织规则

### 总览层

总览层负责快速回答：

- 当前能否继续
- 有多少问题
- 问题主要分布在哪些分类

建议包含：

- 总状态标签
- 问题计数
- 核心动作入口

### 分类层

分类层负责降低信息压力。

建议：

- 按主题分组
- 每组显示摘要和状态
- 默认只展开需要处理的分类

### 明细层

明细层负责解释具体问题。

每个项目建议至少包含：

- 标题
- 详情说明
- 严重程度
- 修复建议

## 默认展开策略

为降低认知负担，建议默认规则为：

- 全通过分类：折叠
- 仅提醒分类：折叠
- 含未通过分类：展开

补充规则：

- 用户手动展开后，在当前会话内保留展开状态
- 通过项默认隐藏，只在用户主动查看时展开

## 大数量场景策略

当一个页面可能出现很多状态项时，应避免一次性全部平铺。

建议做法：

- 默认只展示问题项
- 通过项以数量摘要代替
- 单分类问题过多时只显示前几条
- 提供“查看更多”而不是初始全量展开

如果需要筛选，应保持轻量，不做复杂组合式筛选。

## 轻量筛选建议

优先使用最少但最有效的筛选选项：

- `全部`
- `仅问题`

这里的“仅问题”指：

- `提醒`
- `未通过`

这可以覆盖绝大多数定位需求，同时保持学习成本很低。

## 轻量导航建议

对于问题项较多的页面，建议加入轻量导航：

- `上一条`
- `下一条`

导航范围跟随当前筛选结果变化。

导航应自动完成：

- 切换当前聚焦项
- 滚动定位到对应项
- 更新对应的视觉高亮

## 视觉元素映射建议

同一状态在不同载体上，应有一致映射关系。

建议覆盖以下载体：

- 状态标签
- 图标
- 边框
- 背景浅色
- 文本强调
- 预览标注
- 主按钮状态

例如：

- `通过`：低强度绿色标签和摘要
- `提醒`：中强度琥珀色标签和标注
- `未通过`：高强度红色标签、描边和阻塞态按钮

## 预览或画布中的反馈规则

当页面包含预览区、画布或空间布局时，视觉反馈应注意以下原则：

### 只突出当前关注点

不要默认同时高亮全部问题，否则预览会变成噪音层。

### 类型区分明确

不同问题应尽量有不同标注方式，例如：

- 越界：边缘裁切提示
- 重叠：交叠区域蒙层
- 空内容：虚线框与角标
- 全局问题：不在画布上画框

### 列表与标注要联动

- hover 列表项 -> 高亮预览标注
- click 列表项 -> 锁定预览标注
- click 标注 -> 定位列表项

## 文案规范建议

状态文案应尽量避免模糊词。

建议结构：

- 状态
- 问题名称
- 原因
- 建议动作

示例：

- `未通过：未选择打印机`
- `原因：当前没有目标设备，无法执行打印`
- `建议：先选择在线打印机，再重新检查`

这种结构比只写“打印机异常”更易懂。

## 动作联动规则

反馈应尽量指向最近的修复入口。

建议：

- 设备问题 -> 跳转到设备或校准入口
- 文档问题 -> 跳转到文档规格或编辑器
- 内容问题 -> 定位具体元素或返回编辑区
- 保存问题 -> 直接提供保存动作

## 可复用的数据结构建议

如果后续要在多个模块共享同一套视觉反馈系统，建议抽象为统一结构：

```ts
type FeedbackStatus = 'pass' | 'warn' | 'fail'

type FeedbackLevel = 'page' | 'section' | 'item'

type FeedbackItem = {
  id: string
  status: FeedbackStatus
  title: string
  detail?: string
  actionHint?: string
  category?: string
  blocksPrimaryAction?: boolean
}
```

后续不同模块可以在这套基础上补充自己的定位字段和业务字段。

## 在 README 中的推荐表达方向

如果后续将这套规则写入 README，建议保留以下内容：

- 为什么要统一视觉反馈
- 三档状态的定义
- 默认通过收起、问题优先的设计原则
- 大数量场景下的轻量筛选与导航原则
- 列表与预览联动的设计原则

不建议在 README 中写入过细的界面实现细节或临时字段名。

## 适用范围

这套常规视觉反馈系统后续可用于：

- 打印检查
- 模板保存与离开提醒
- 导出校验
- 打印机与设备状态
- 词库内容校验
- 批量操作结果反馈

## 后续演进方向

后续可在不破坏主结构的前提下扩展：

- 更丰富的分类体系
- 更细的可视化标注类型
- 可访问性优化
- 键盘导航
- 多模块统一的状态令牌和组件规范

但无论如何扩展，都应保持这几个基本原则不变：

- 语义一致
- 问题优先
- 默认低负担
- 反馈可行动
