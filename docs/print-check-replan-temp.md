# 打印检查临时改造方案与计划

## 背景

当前打印检查已经具备基础清单和预览能力，但还缺少一套完整、统一、低负担的视觉反馈体系。

现阶段的主要问题不是“检查项不够多”，而是：

- 检查结果缺少统一的状态语义。
- 问题列表、预览区标注、顶部总状态之间关联不够强。
- 检查项数量一多时，用户会被几十条混杂的 `通过 / 提醒 / 未通过` 信息淹没。
- 预览区缺少稳定的问题定位和反向导航。

这次改造的目标不是先增加更多规则，而是先把“怎么呈现问题、怎么组织问题、怎么帮助用户快速定位并修复问题”规划清楚。

## 改造目标

打印检查页需要优先回答三个问题：

- 能不能打印。
- 先改什么。
- 问题在哪。

因此首要目标是建立一套统一的：

- 检查项数据结构。
- 状态分级体系。
- 分类与折叠规则。
- 清单与预览的双向联动规则。
- 轻量筛选与轻量导航规则。

## 总体设计原则

### 1. 问题优先，不做结果堆叠

打印检查页不应默认平铺所有检查结果，而应优先突出：

- 未通过项。
- 提醒项。
- 当前聚焦的问题。

通过项默认收起或弱化显示，不与问题项抢注意力。

### 2. 总览优先，细节按需展开

用户进入检查页后，第一眼应先看到：

- 当前总状态。
- 未通过数量。
- 提醒数量。
- 哪些分类需要处理。

只有当用户进入某个分类后，才继续展开具体项目。

### 3. 预览区保持克制

预览区的目标是辅助定位，而不是把所有问题同时铺在画面上。

默认只高亮：

- 当前 hover 的问题。
- 当前选中的问题。
- 或当前聚焦分类中的有限标注。

避免预览区出现满屏红黄框导致噪音过高。

### 4. 联动必须双向可达

问题清单和预览标注不能各自独立，必须支持：

- 从问题项定位到预览。
- 从预览标注反向定位到问题项。
- 分类切换时切换对应标注上下文。

## 视觉状态体系

统一采用三档状态：

- `通过`
- `提醒`
- `未通过`

建议语义如下：

### 通过

- 含义：条件满足，不影响打印。
- 视觉方向：绿色系。
- 默认展示策略：不默认逐条展开，只在分类摘要中显示数量。

### 提醒

- 含义：允许打印，但存在风险、隐患或建议处理项。
- 视觉方向：琥珀/黄色系。
- 默认展示策略：默认折叠分类，但在问题分类区域靠前显示。

### 未通过

- 含义：当前不允许直接打印，属于阻塞项。
- 视觉方向：红色系。
- 默认展示策略：默认展开，并优先占据视线。

## 信息架构

打印检查页建议固定成三部分：

### 顶部总览条

负责快速传达整体判断。

建议包含：

- 总状态：`可打印 / 建议处理后打印 / 暂不可打印`
- 数量摘要：`未通过 X`、`提醒 Y`
- 轻量筛选
- 轻量导航

### 预览区

负责可视化定位问题。

建议只做：

- 当前标签输出预览。
- 当前问题标注高亮。
- 点击标注反向定位到问题项。

### 分类检查区

负责组织问题和承载修复入口。

建议用分类卡片承载问题，而不是单一长列表。

## 分类方案

首版建议固定为以下 6 类：

- 设备与连接
- 打印校准
- 文档规格
- 内容完整性
- 布局与重叠
- 输出准备

每个分类显示摘要信息：

- 分类名称
- 分类状态
- `未通过 / 提醒 / 通过` 数量
- 是否影响打印
- 展开或折叠状态

分类状态由子项聚合得出：

- 只要存在 `未通过`，分类状态即为 `未通过`
- 没有 `未通过` 但有 `提醒`，分类状态为 `提醒`
- 全部通过，则分类状态为 `通过`

## 默认展开与折叠规则

这部分是降低认知负担的核心。

### 默认规则

- 分类下全部 `通过`：默认折叠
- 分类下只有 `提醒`：默认折叠
- 分类下存在 `未通过`：默认展开

### 交互规则

- 用户点击某个分类时，可手动展开或折叠
- 当前会话内应记住用户手动展开状态
- 切换筛选或切换当前问题时，不应频繁重置用户的手动选择

### 通过项策略

- 分类默认不逐条显示通过项
- 分类头只显示“已通过 N 项”
- 用户主动点“查看通过项”后，再展开通过项明细

## 分类内部展示规则

分类展开后，不应将所有状态混排。

建议排序：

- 先 `未通过`
- 再 `提醒`
- 最后 `通过`

同级内进一步按这类顺序排序：

- 阻塞打印的问题
- 影响打印质量的问题
- 一般性提示

若单分类问题很多，建议：

- 默认只显示前 `5` 条问题项
- 其余通过“查看更多”展开

这样可以避免一个分类展开后直接塞入十几条内容。

## 轻量筛选方案

筛选需要保留，但不能做重。

首版建议只做两个选项：

- `全部`
- `仅问题`

其中：

- `全部` = `通过 + 提醒 + 未通过`
- `仅问题` = `提醒 + 未通过`

默认建议为：

- 进入检查页时默认使用 `仅问题`

这样用户一进来先处理问题，不会先看到大量通过项。

## 轻量导航方案

导航也需要保留，但保持极简。

首版建议只做：

- `上一条问题`
- `下一条问题`

导航范围跟随当前筛选结果变化。

例如当前筛选为 `仅问题` 时：

- 导航仅在 `提醒 + 未通过` 范围内循环

导航触发时需要联动：

- 右侧滚动到目标问题项
- 自动展开对应分类
- 预览区高亮目标标注

## 预览联动规则

预览区和问题清单之间建立双向关联。

### 从问题项到预览

- hover 问题项：预览区临时高亮对应标注
- click 问题项：预览区锁定高亮对应标注
- 切换分类：预览区切换当前分类上下文

### 从预览到问题项

- hover 标注：右侧对应问题项高亮
- click 标注：右侧滚动到问题项并展开详情

### 全局问题

对于没有具体画布位置的问题，例如：

- 未选择打印机
- 打印机离线
- 当前文档未保存

建议：

- 在问题清单中显示
- 在预览区不画区域框
- 可用“全局问题”标签代替坐标标注

## 预览标注规范

不同问题不应全部使用同一种红框。

建议分型如下：

### 元素越界

- 使用明显边框或裁切线提示
- `提醒` 和 `未通过` 在样式强度上区分

### 元素重叠

- 使用相交区域半透明蒙层
- 条码/二维码重叠要比普通文本重叠更强提示

### 空文本、空条码、空二维码

- 使用虚线框和问题角标

### 全局状态类问题

- 不在预览区画框
- 在预览头部或右侧清单中表达

### 当前聚焦问题

- 在同类标注上增加更高层级的高亮态
- 避免用户分不清“当前正在看的问题”和“其它问题”

## 检查项数据结构建议

建议将打印检查项统一为一套结构，所有 UI 都从同一份数据驱动。

```ts
type CheckStatus = 'pass' | 'warn' | 'fail'

type CheckCategory =
  | 'device'
  | 'calibration'
  | 'document'
  | 'content'
  | 'layout'
  | 'output'

type CheckTarget =
  | { kind: 'global' }
  | { kind: 'element'; elementId: string }
  | { kind: 'region'; left: number; top: number; width: number; height: number }

type PrintCheckItem = {
  id: string
  category: CheckCategory
  status: CheckStatus
  title: string
  detail: string
  actionHint?: string
  target: CheckTarget
  blocksPrinting: boolean
  sortOrder: number
}
```

补充建议：

- `title` 用于列表标题
- `detail` 用于解释具体问题
- `actionHint` 用于提示修复入口
- `target` 用于驱动预览联动
- `blocksPrinting` 用于按钮状态判定

## 前端 UI 状态建议

除了检查项本身，还需要一组轻量 UI 状态来承载分类和联动逻辑。

```ts
type CheckFilter = 'all' | 'issues'

type PrintCheckUiState = {
  filter: CheckFilter
  activeItemId: string | null
  expandedCategories: Record<string, boolean>
  showPassedItemsByCategory: Record<string, boolean>
}
```

首版不建议引入复杂状态机，先保持状态足够轻。

## 交互规则建议

### 总状态与打印按钮

- 存在 `未通过`：禁止直接打印
- 没有 `未通过` 但有 `提醒`：允许打印，但明确提示
- 全部 `通过`：正常允许打印

### 问题项信息结构

每个问题项建议固定包含三段：

- 问题名称
- 为什么会影响打印
- 下一步建议

例如：

- `未通过：条码内容为空`
- `原因：空条码会导致打印结果不可扫描`
- `建议：返回编辑器，为该条码填写内容后重新检查`

### 修复入口绑定

不同类型的问题应尽量给最近路径：

- 设备与连接 -> 打开打印校准或切换打印机
- 文档规格 -> 打开文档规格
- 内容问题 -> 返回编辑器
- 保存状态 -> 立即保存

## 首版实施边界

首版建议先做以下范围：

- 统一检查项结构
- 增加分类卡片
- 增加默认折叠规则
- 增加 `全部 / 仅问题` 轻量筛选
- 增加 `上一条 / 下一条问题` 轻量导航
- 增加问题项与预览的单点联动

首版暂不追求：

- 多维复杂筛选
- 搜索问题项
- 批量修复
- 同时显示全部问题标注
- 面向高级用户的审计型数据表

## 建议实施顺序

### 第一阶段：数据模型统一

- 重构打印检查输出结构
- 为检查项补齐分类、目标、排序和动作提示
- 统一 `通过 / 提醒 / 未通过` 枚举和聚合逻辑

### 第二阶段：分类结构与默认折叠

- 引入分类摘要卡片
- 实现默认展开与折叠策略
- 将通过项从默认明细中移出

### 第三阶段：轻量筛选与轻量导航

- 增加 `全部 / 仅问题`
- 增加 `上一条 / 下一条问题`
- 让导航与当前筛选范围保持一致

### 第四阶段：预览联动

- 实现问题项 hover 和 click 联动
- 实现标注到问题项的反向定位
- 控制预览区默认只突出当前聚焦问题

### 第五阶段：文案与视觉打磨

- 统一状态文案和动作提示
- 优化分类摘要、问题卡片和标注层级
- 验证大数量场景下的信息负担

## 验证重点

需要重点验证以下场景：

- 仅有 1-2 个问题时是否足够直接
- 同时存在十几条提醒时是否仍然清晰
- 某一个分类下存在大量问题时是否能被有效折叠
- 全通过场景是否足够轻量
- 预览区是否因标注过多而噪音过高
- 用户能否快速跳到下一个问题并理解修复路径

## 后续沉淀方向

这份文档是临时改造方案，后续应将其中更稳定的部分抽离为：

- README 中的打印检查设计原则
- 项目的通用视觉反馈系统规范
- 组件级的状态语义与交互约定
