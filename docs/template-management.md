# 模板功能设计规则

> 状态：当前设计参考。
> 当前用途：保留模板/草稿边界、本地模板行为、基础 API 规则和验收标准。

## 结论

当前模板功能不适合继续在原结构上小修小补，应按“模板库 + 编辑草稿 + 导入导出”三条边界重做。

第一版目标不是做完整模板平台，而是让本地打印工作流稳定：

- 能看到有哪些模板
- 能打开模板变成可编辑草稿
- 能保存当前草稿为模板
- 能覆盖保存已打开模板
- 能复制、重命名、删除模板
- 模板文件可读、可备份、可迁移

## 当前问题

### 职责混在一起

现在模板同时承担：

- 模板库记录
- 当前编辑文档
- 多标签页打开状态
- 脏状态判断
- 导入来源
- 元信息编辑
- 词库绑定
- 打印参数

结果是“模板”和“正在编辑的草稿”边界不清。用户打开模板后，很难判断自己是在改模板本体，还是在改一个临时副本。

### 保存语义不清

当前有“保存”和“保存为模板”，但实际状态取决于 `templateId`、`lastSavedSnapshot`、tab 状态和当前来源。这个模型过于隐式，容易出现：

- 新建标签不知道保存到哪里
- 打开模板后修改，用户不知道是否会覆盖原模板
- 复制模板和另存为语义接近但状态不同
- 删除模板后已打开内容还保留，绑定关系又被打断

### 模板数据过早承载高级能力

模板文件里已经有 `description`、`tags`、`source`、`lastUsedAt`、词库绑定等扩展点。它们本身不坏，但在基础保存/打开不稳定时，会让问题面变大。

第一版应该把模板当成“可命名的标签版式文件”，高级管理能力后置。

### UI 入口过多

模板库、顶部保存按钮、右侧模板属性、导入 DDL、多标签页、删除后的草稿保留同时存在。入口多，但用户心智没有被收敛。

## 新模型

### 核心对象

#### Template

模板是持久化文件。

模板包含：

- `id`
- `name`
- `createdAt`
- `updatedAt`
- `document`

第一版可选保留：

- `description`
- `tags`

暂缓：

- `source`
- `lastUsedAt`
- 复杂分类
- 模板市场
- 模板版本历史

#### Draft

草稿是当前正在编辑的工作副本。

草稿包含：

- `draftId`
- `templateId`
- `document`
- `savedSnapshot`
- `dirty`

规则：

- 打开模板时，复制一份 `document` 到草稿
- 编辑只改草稿，不直接改模板文件
- 点“保存”才写回模板文件
- 点“另存为”创建新模板，并把当前草稿绑定到新模板
- 新建空白标签一开始没有 `templateId`

#### Library

模板库只负责列表和文件操作。

模板库不理解编辑器多标签状态，不负责判断草稿是否脏。

## 第一版用户流程

### 打开模板

1. 用户进入模板库
2. 单击模板卡片
3. 系统创建或切换到该模板对应草稿
4. 状态显示 `已打开模板：xxx`

如果同一模板已经打开：

- 默认切换到已有草稿
- 不重复打开第二份

### 保存

有 `templateId`：

- 覆盖保存当前模板
- 更新 `updatedAt`
- 更新草稿 `savedSnapshot`

没有 `templateId`：

- 触发“另存为”

### 另存为

1. 输入模板名称
2. 创建新模板文件
3. 当前草稿绑定到新模板
4. 模板库刷新

### 复制模板

复制是模板库动作，不影响当前草稿，除非用户选择“复制并打开”。

第一版建议只做：

- 复制
- 复制成功后模板库出现新模板

暂不自动打开复制件，减少状态跳跃。

### 重命名

重命名只修改模板元信息。

如果该模板已打开：

- 同步更新草稿标题
- 不改变草稿内容
- 不重置 dirty 状态

### 删除

删除是危险操作。

第一版规则：

- 删除前二次确认
- 如果模板已打开，删除成功后该草稿变成未绑定草稿
- 草稿保留，避免用户误删后丢失当前画布

## 数据格式

### 内部模板文件

```json
{
  "schemaVersion": 1,
  "id": "shipping-hello-20260705031926",
  "name": "Shipping Hello",
  "description": "",
  "tags": [],
  "createdAt": "2026-07-05T03:19:26+08:00",
  "updatedAt": "2026-07-05T03:19:26+08:00",
  "document": {
    "name": "Shipping Hello",
    "widthMm": 40,
    "heightMm": 30,
    "copies": 1,
    "darkness": 8,
    "gapMm": 2,
    "elements": []
  }
}
```

### 设计约束

- `document.name` 与模板 `name` 保持一致
- 文件名只作为存储细节，不作为用户可见名称
- 模板读取失败不能拖垮整个模板库
- 读取失败的模板应在诊断里可见，后续再做修复入口

## 本地模板存储规则

保留文件存储，但简化接口：

- `ListAsync`
- `GetAsync`
- `CreateAsync`
- `SaveAsync`
- `RenameAsync`
- `DuplicateAsync`
- `DeleteAsync`

调整点：

- 增加明确的 normalize/migrate 层
- `DeleteAsync` 只按明确 id 删除单个文件
- 所有写入走临时文件再替换，避免半写坏文件

## API 行为

建议接口：

- `GET /api/templates`
- `GET /api/templates/{id}`
- `POST /api/templates`
- `PUT /api/templates/{id}`
- `PATCH /api/templates/{id}/name`
- `POST /api/templates/{id}/duplicate`
- `DELETE /api/templates/{id}`

暂不做：

- 单独 meta patch 多字段
- last used
- 复杂搜索服务端排序

搜索和排序第一版放前端即可。

## 验收标准

模板功能第一版只要满足以下标准就算可用：

- 新建一个标签后，可以保存为模板
- 关闭再打开应用后，模板仍在
- 打开模板修改后，保存能覆盖原模板
- 另存为不会覆盖原模板
- 删除模板不会丢失当前已打开草稿
- 一个坏模板文件不会导致整个模板库空白或崩溃
- 用户在任何时刻都能看出当前内容是否未保存

## 暂缓功能

这些功能先不要做：

- 模板市场
- 在线同步
- 会员模板
- 模板版本历史
- 批量模板导入导出
- 模板分类系统
- 模板变量系统
- 自动保存
- 最近使用排序

## 后续方向

基础可用后，再按顺序加：

1. `.yblabel.json` 导入导出
2. 模板说明和标签
3. 模板缩略图
4. 模板修复/诊断入口
5. 最近使用
6. 模板版本备份
