# 模板编辑器词库动作临时实施计划

更新日期：2026-07-20

## 目标

按 [template-lexicon-actions-ux.md](E:/Program/YiboLabel/docs/template-lexicon-actions-ux.md) 的定稿方案，完成模板编辑器词库动作的 UX 收口与实现。

本计划只服务本轮施工，不作为长期产品设计文档替代品。

## 实施范围

覆盖：

- 词库页底部两按钮动作区
- 拖放前目标判定与动作提示
- 拖放成功后的动作记录
- 右按钮的最近词库动作撤回
- 动作通知统一

不覆盖：

- 词库页整体大改版
- 多选批量词库动作
- 大范围视觉重画
- 历史系统重构

## 施工原则

1. 先统一动作模型，再改 UI。
2. 先保证拖放与按钮共用同一套动作记录，再接右按钮撤回。
3. 不为了打通状态而强制改变当前视觉选中元素。
4. 禁止把“最近拖放目标元素”混入左按钮判断。

## 分阶段计划

### 第一阶段：收拢动作状态

目标：

- 明确区分当前选择元素、最近拖放目标元素、最近一次词库动作记录

任务：

- 梳理当前模板编辑器中词库相关状态
- 新增或整理最近一次词库动作记录结构
- 明确动作来源字段：按钮或拖放
- 确保绑定、解绑、应用都能写入统一动作记录

完成标准：

- 右按钮后续可仅依赖最近动作记录运转

### 第二阶段：重做左按钮规则

目标：

- 左按钮只基于当前选择元素和当前词库选择判断

任务：

- 固定左按钮为单槽位
- 接入三种可用动作：
  - 绑定分组
  - 取消绑定
  - 应用条目
- 在无效状态下禁用按钮
- 文案保持上一次动作名，直到下一次可用状态切换时再更新

完成标准：

- 左按钮不再受拖放目标干扰
- 左按钮不显示长提示文案

### 第三阶段：重做右按钮规则

目标：

- 右按钮只服务最近一次词库动作撤回

任务：

- 固定右按钮为单槽位
- 根据最近动作类型切换文案：
  - 撤消绑定
  - 恢复绑定
  - 撤消应用
- 实现对应的撤回逻辑
- 确保不依赖当前仍选中原元素或原分组

完成标准：

- 用户切换选择后，仍能撤回刚才词库动作

### 第四阶段：统一拖放判定与提示

目标：

- 拖放前就能显示将执行的动作或禁止原因

任务：

- 分组拖放规则：
  - 可绑定目标显示 `绑定分组`
  - 非法目标显示 `不可绑定：该元素不支持词库`
- 条目拖放规则：
  - 文本：`应用到文本`
  - 二维码：`应用到二维码`
  - 条码：按 Code128 规则校验
  - 非内容元素：`不可应用：该元素不支持文本`
- 保持拖放成功后不自动切换当前视觉选中元素

完成标准：

- 用户能在落下前理解动作结果
- 拖放不再破坏连续操作节奏

### 第五阶段：统一通知与收尾

目标：

- 弱化按钮副文案，强化动作完成后的结果反馈

任务：

- 统一绑定、解绑、恢复绑定、应用、撤消应用的通知文案
- 清理旧的局部撤消按钮残留逻辑
- 检查拖放与按钮是否都走统一动作记录

完成标准：

- 同一动作无论来自按钮还是拖放，通知与撤回行为一致

## 重点风险

### 风险 1：左按钮误读拖放目标

如果把最近拖放目标元素混入左按钮判断，会再次把“当前上下文动作”和“最近动作上下文”搅在一起。

要求：

- 左按钮严格只看当前选择元素

### 风险 2：右按钮回退依赖当前选择

如果回退仍要求用户保持原选择，UX 会再次退回旧问题。

要求：

- 右按钮严格只看最近一次词库动作记录

### 风险 3：拖放成功后自动改选中

这会在连续大量拖放时制造明显视觉跳动。

要求：

- 拖放成功后不自动切换当前视觉选中元素

### 风险 4：绑定与应用动作记录不完整

如果动作记录缺少必要字段，右按钮会无法稳定撤回。

要求：

- 记录元素、分组、条目、前值、后值和动作来源

## 验收清单

1. 左按钮是否只基于当前选择元素和当前词库选择判断
2. 右按钮是否能在切换选择后仍正确撤回刚才动作
3. 拖放成功后是否未自动改变当前视觉选中元素
4. 拖放前是否能正确显示动作提示或禁止原因
5. 条码拖放是否按 Code128 规则前置拦截
6. 按钮与拖放是否都写入统一最近动作记录
7. 动作完成通知是否统一

## 施工顺序建议

1. 状态与动作记录
2. 左按钮
3. 右按钮
4. 拖放提示
5. 通知与收尾

## 备注

如果实现过程中发现旧逻辑仍存在不合理处，应继续主动提出并在不破坏本定稿边界的前提下讨论，不默认沿用旧设计。
